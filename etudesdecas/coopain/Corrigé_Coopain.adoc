== Mission 1 : Gestion des demandes d'insémination

**Question 1.1 : Règle de gestion (Requête SQL)**

La sous-requête avec `MIN(date)` et la condition `stockRestant >= 1` permettent de déduire la règle suivante : "Lors d'une demande d'insémination, le système sélectionne automatiquement le lot de paillettes **le plus ancien** (règle du Premier Entré, Premier Sort - FIFO) pour le taureau et le type de paillette choisis, à condition qu'il reste **au moins une paillette en stock** dans ce lot."

**Question 1.4 : Requête des stocks par taureau**
[source,sql]
----
SELECT T.idNational, T.nom, SUM(L.stockRestant) AS stockTotal
FROM Taureau T
JOIN Lot L ON T.idNational = L.idNationalTaureau
GROUP BY T.idNational, T.nom
ORDER BY stockTotal ASC;
----

=== Partie 2 : Audit de sécurité (DICP et Mobilité)
**Question 2.3 : Perte du mobile (Risques et Solutions)**

* **Confidentialité** : L'attaquant peut lire les données SQLite en clair (informations sur les adhérents, les tournées, les véhicules).
* **Intégrité** : L'attaquant peut modifier la base locale (`kmFin`) avant la prochaine synchronisation, faussant ainsi les frais de déplacement.
* **Solutions** : Mise en place d'un **MDM** (Mobile Device Management) pour effacer l'appareil à distance (Remote Wipe), obligation d'un code de déverrouillage fort, et utilisation de **SQLCipher** pour chiffrer le fichier de base de données local avec une clé matérielle.

**Question 2.4 : Vulnérabilité OWASP (Mot de passe)**
Stocker ou manipuler un mot de passe en clair dans un attribut d'objet (`private String password;`) expose l'application à des fuites lors d'un dump mémoire, ou à l'interception si cet objet est sérialisé et transmis sur le réseau.

**Recommandation** : Ne jamais stocker le mot de passe en clair. L'application doit utiliser des mots de passe hachés (ex: `Bcrypt`), ou mieux, utiliser un système de jetons d'authentification éphémères de type **JWT (JSON Web Token)**.


== Mission 3 : Gestion des tournées (POO)

**Question 3.1.1 : Mécanisme de différenciation (Surcharge)**
Il s'agit du mécanisme de **surcharge (overloading)** en Programmation Orientée Objet. Bien que les méthodes aient le même nom (`CATournee`), le compilateur Java les différencie par leur signature, c'est-à-dire le nombre et le type de leurs paramètres (l'une n'a pas de paramètre, l'autre prend un objet `TypePrestation`).

public ArrayList<Adherent> getAdherents() {
    ArrayList<Adherent> listeAdherents = new ArrayList<>();

    for (Visite v : this.laTournee.getLesVisites()) {
        Adherent a = v.getLeAdherent();

**Question 3.3 : Cybersécurité des API (L'attaque MitM)**

Une attaque **Man-in-the-Middle (MitM)** consiste pour un pirate à s'intercaler entre le smartphone de l'inséminateur et le serveur Web de l'API (par exemple en créant un faux point d'accès Wi-Fi public). Il peut alors lire les données de synchronisation en clair.
Pour contrer cela, le serveur doit impérativement utiliser le protocole **HTTPS** (couche de chiffrement **TLS/SSL**) afin de chiffrer le tunnel de communication de bout en bout.

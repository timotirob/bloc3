== Présentation du contexte

=== L’organisation cliente
Puces’In est une entreprise française fondée sur le modèle de l’économie circulaire qui récupère, collecte, rachète et revend des livres d’occasion. Puces’In dispose de sept centres de dépôt, tri et revente répartis sur tout le territoire national.

L'entreprise se différencie par deux applications majeures :

* **Puces’In Rachète** : Application mobile (Android/iOS) permettant aux particuliers de scanner les codes-barres de leurs livres pour les revendre.
* **VerifColis** : Application client lourd (C#) utilisée dans les centres de tri pour vérifier le contenu des colis reçus.

Les livres sont ensuite revendus en rayon ou via le site web **pucesin.com**.

=== L’entreprise prestataire de service
La société **Solut** est une ESN labellisée ExpertCyber. Vous intervenez en tant que jeune recrue sur la sécurisation des applications et la résolution d’incidents.

== DOSSIER DOCUMENTAIRE

==== Document commun 1 : Processus de rachat de livres
Un livre est un exemplaire d’une publication. Chaque publication de livre est caractérisée par un numéro d'identification unique ISBN (International Standard Book Number), un titre, un auteur, une date de publication, un type de publication (poche, broché, bande dessinée, manga, etc.) et un prix de vente neuf.

Pour chaque publication d'un livre, plusieurs prix de revente sont définis, en fonction de l’état du livre (son niveau d’usure), et de la demande liée à la publication. Par exemple, les mangas étant plus demandés que les livres de poche, ils sont revendus plus chers.

Le processus de rachat de livres est le suivant :

1. Le particulier qui souhaite revendre des livres à Puces’In doit lire le code-barres de chacun d'eux avec l’application **Puces’In Rachète** qu’il aura préalablement installée.

2. L’application affiche le prix de rachat estimé pour chaque livre. Ce prix correspond à **10 % du prix de revente** d'un livre en « bon état » à la clientèle (un prix de revente est défini pour chaque publication de livre et chaque état).
* _Exemple : si le prix de revente d'une publication pour un livre en « bon état » est fixé à 10 €, alors le prix de rachat estimé d'un livre appartenant à cette publication sera de 1 €._

3. Une fois que le vendeur a traité tous les livres qu’il souhaite revendre, il valide son panier.

4. L'application lui propose alors de se connecter, ou de créer un compte. Dans ce dernier cas, son adresse postale ainsi que l’identifiant international de son compte bancaire (**IBAN**) lui seront demandés.

5. Une fois que le panier est validé, un bordereau d’expédition (appelé également étiquette d'envoi) est généré pour le vendeur, qui peut alors l’imprimer et le coller sur le colis contenant les livres du panier. Le bordereau d'expédition contient un code-barres permettant d'identifier le colis.

6. Le client expédie le colis à Puces’In qui le réceptionnera dans son centre de tri.

7. Les frais de port seront réglés par Puces'In dès que le client aura déposé le colis au bureau de poste ou au point relais de son choix.

8. À la réception du colis, l'état du colis est mis à jour dans la base de données et passe à l'état **« reçu »**.

9. Puis, un employé du centre de tri est chargé de renseigner l’état de chaque livre présent dans le colis. Cet état peut prendre une des valeurs suivantes : « comme neuf », « très bon état », « bon état », « acceptable », « refusé ». Si le livre manque dans le colis, alors il prendra l'état **« absent »**.

10. Si un livre est dans un état **« acceptable »**, alors l’employé du centre de tri saisira une nouvelle proposition de prix de rachat, enregistrée dans le champ `prixRachatPropose`, et l’enverra au client par messagerie. Si le client ne valide pas cette proposition dans un délai d'un mois, alors le livre sera retourné au client.

11. Si un livre est dans un état **« refusé »**, alors il sera également retourné au client.

12. Le client sera réglé :
* Immédiatement si dans le colis tous les livres ont un état supérieur à l’état « acceptable » (« comme neuf », « très bon état » ou en « bon état ») ;
* Ou suite à l'acceptation du client de la nouvelle proposition de prix de rachat si un des livres du colis est dans un état « acceptable ».

13. Les livres sont ensuite nettoyés.

14. Le service comptabilité de Puces’In procède alors au virement du montant à régler au client à l’aide d’un logiciel dédié.

[.underline]#Traitement du formulaire :#
Au clic sur le bouton « Valider le colis » la méthode traiterUnLivre fournie ci-dessous est appelée pour chaque ligne de la grille de données.
Elle reçoit deux paramètres :

* un objet unLivre correctement valorisé avec les valeurs du livre d’une ligne et
* un objet contenant les caractéristiques utiles à la connexion à la base de données.


[source,csharp]
----
private int traiterUnLivre(Livre unLivre, MySqlConnexion uneConnexion) {
int valeurRetournee;
string reqSQL;

// test si le livre est dans l'état « acceptable »
if (unLivre.getLeEtatLivre().getId() == 4)
{
// mise en forme de la requête SQL
// les paramètres de la requête sont précédés du caractère @
reqSQL = "update Livre set idEtat = @pEtat, prixRachatPropose = @pPrix where numLivre = @pLivre and idPublication= @pPub";
}
else
{
// mise en forme de la requête SQL
reqSQL = "update Livre set idEtat = @pEtat where numLivre = @pLivre and idPublication= @pPub";
}

// mise en forme de la requête SQL dans l'objet uneComSql
MySqlCommand uneComSql = new MySqlCommand(reqSQL, uneConnexion);

// renseigne les paramètres à partir des valeurs de l'objet unLivre
uneComSql.Parameters.AddWithValue("@pEtat", unLivre.getLeEtatLivre().getId());

if (unLivre.getLeEtatLivre().getId() == 4)
{
// si l'état du livre est acceptable, le prix de rachat proposé saisi est enregistré
uneComSql.Parameters.AddWithValue("@pPrix", unLivre.getPrixRachatPropose());
}

uneComSql.Parameters.AddWithValue("@pLivre", unLivre.getNumLivre());
uneComSql.Parameters.AddWithValue("@pPub", unLivre.getLaPublication().getId());

// Exécution de la requête contenue dans l'objet uneComSql
// la méthode ExecuteNonQuery retourne le nombre d'enregistrements modifiés
valeurRetournee = uneComSql.ExecuteNonQuery();

return valeurRetournee;

=== Document A6 : Composition d'un jeton JWT
JWT signifie JSON Web Token : c’est un standard ouvert (RFC 7519) qui décrit une méthode pour transmettre une information de manière sécurisée.

Les jetons JWT sont utilisés comme jetons d’authentification pour consommer un service. Ils sont généralement placés dans l’entête d'un échange HTTP du type autorisation de porteur (Bearer).

Un jeton JWT est composé de trois parties séparées par un point « . » :

L'entête (header) contient des informations sur le jeton (type de jeton et algorithme utilisé pour la signature) au format JSON encodées en base 64.

Le corps appelé également charge utile (payload) au format JSON encodé en base 64 contient des informations utiles au serveur utilisant le jeton.

La signature du jeton est un condensé (hash) obtenu en appliquant une fonction de hachage sur l'entête concaténé à la charge utile. Cette fonction utilise une clé secrète pour s'assurer de la validité du jeton.

image::structure_jwt.png[Schéma de la structure Header.Payload.Signature]

[.underline]#Détail de la charge utile (payload) :#

Les informations sont fournies sous la forme "nomDePropriété" : valeur. Voici des exemples de propriétés (claims) définies par le standard :

* iss (Issuer) : identifie le serveur qui a émis le jeton.
* iat (Issued At) : date de génération du jeton au format timestamp.
* exp (Expiration Time) : date d'expiration du jeton au format timestamp.

[.underline]#Exemple de décodage d'un jeton JWT :#

Voici un exemple de jeton transmis lors d'un échange :
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJpc3MiOiJodHRwczovL3B1Y2VzaW4uY29tIiwiaWF0IjoxNTIxMzA5NjAwLCJleHAiOjE1MjEzMTMyMDB9.sGrqR9WhhjDvs6zH56IVemEVNvmJs63ic2yhQOJJUyU

[cols="1,2", options="header"]
|===
| Partie | Contenu décodé (JSON)
| Entête | { "typ": "JWT", "alg": "HS256" }
| Charge utile | { "iss": "https://pucesin.com", "iat": 1521309600, "exp": 1521313200 }
|===

[.underline]#Calcul de la signature :#

La signature est obtenue en utilisant l'algorithme HS256 (HMAC-SHA-256) de la manière suivante :

Signature = HMACSHA256(base64UrlEncode(header) + "." + base64UrlEncode(payload), clé_secrète)

La clé secrète est celle utilisée par le serveur pour générer la signature du jeton

<?php
header('Content-Type: application/json');
require_once 'config.php';
use \Firebase\JWT\JWT;

$data = json_decode(file_get_contents("php://input"));

if (!empty($data->login) && !empty($data->password)) {
    $db = getDB();
    $stmt = $db->prepare("SELECT * FROM Utilisateur WHERE login = ?");
    $stmt->execute([$data->login]);
    $user = $stmt->fetch();

    if ($user && password_verify($data->password, $user['password_hash'])) {
        $payload = [
            "iss" => "pucesin-api",
            "iat" => time(),
            "exp" => time() + 3600, // 1 heure
            "uid" => $user['id'],
            "role" => $user['role']
        ];

        $jwt = JWT::encode($payload, JWT_SECRET, JWT_ALGO);
        echo json_encode(["token" => $jwt, "message" => "Succès"]);
    } else {
        http_response_code(401);
        echo json_encode(["message" => "Identifiants invalides"]);
    }
}

header('Content-Type: application/json');
require_once 'config.php';
use \Firebase\JWT\JWT;
use \Firebase\JWT\Key;

$authHeader = $_SERVER['HTTP_AUTHORIZATION'] ?? '';

if (preg_match('/Bearer\s(\S+)/', $authHeader, $matches)) {
    $jwt = $matches[1];
    try {
        $decoded = JWT::decode($jwt, new Key(JWT_SECRET, JWT_ALGO));
        $userId = $decoded->uid;

        $db = getDB();
        $stmt = $db->prepare("
            SELECT L.id, P.titre, P.auteur, L.etat, L.prix_rachat_propose
            FROM Livre L
            JOIN PublicationLivre P ON L.idPublication = P.id
            WHERE L.idProprietaire = ?
        ");
        $stmt->execute([$userId]);
        echo json_encode($stmt->fetchAll());

    } catch (Exception $e) {
        http_response_code(401);
        echo json_encode(["message" => "Accès refusé : Jeton invalide ou expiré"]);
    }
} else {
    http_response_code(401);

Le > est la syntaxe de PhpStorm/IntelliJ pour introduire un response handler — c'est lui qui dit "ce bloc de script doit s'exécuter après réception de la réponse".

[source,adoc]

C'est un peu l'équivalent d'un callback : "une fois la réponse reçue >, fais ceci {% ... %}".


**Avantages :**

* Exécution en un clic via la flèche verte dans la marge.
* Historique des requêtes conservé.
* Supporte les variables d'environnement.

== Service de Communication API (Cœur du Prototype)

Cette classe gère l'authentification et le stockage du jeton **JWT** . Elle utilise le `HttpClient` natif de Java.

[.underline]#Fichier : `ApiService.java`#
[source,java]
----
package com.pucesin.service;

import com.fasterxml.jackson.databind.JsonNode;
import com.fasterxml.jackson.databind.ObjectMapper;
import com.pucesin.model.Livre;

import java.net.URI;
import java.net.http.*;
import java.util.List;

public class ApiService {
    private static final String BASE_URL = "http://localhost/pucesin/";
    private static String jwtToken = null;
    private final HttpClient client = HttpClient.newHttpClient();
    private final ObjectMapper mapper = new ObjectMapper();

    // Authentification et récupération du jeton
    public boolean login(String login, String password) throws Exception {
        String json = String.format("{\"login\":\"%s\", \"password\":\"%s\"}", login, password);

        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create(BASE_URL + "login.php"))
                .header("Content-Type", "application/json")
                .POST(HttpRequest.BodyPublishers.ofString(json))
                .build();

        HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());

        if (response.statusCode() == 200) {
            JsonNode node = mapper.readTree(response.body());
            jwtToken = node.get("token").asText();
            return true;
        }
        return false;
    }

    // Récupération des livres avec le Header Authorization
    public List<Livre> getMesLivres() throws Exception {
        if (jwtToken == null) throw new Exception("Non authentifié");

        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create(BASE_URL + "mes_livres.php"))
                .header("Authorization", "Bearer " + jwtToken) // Injection du jeton JWT
                .GET()
                .build();

        HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());

        return mapper.readValue(response.body(),
                mapper.getTypeFactory().constructCollectionType(List.class, Livre.class));
    }
}

== Mission G - Développement de la Logique Métier et Tests Unitaires

=== Contexte de la demande
La direction de **Puces'In** souhaite automatiser le calcul des offres de rachat. Le système doit désormais appliquer une règle de "Bonus Qualité" gérée directement par l'application cliente avant l'affichage final.

=== Objectifs techniques
* Implémenter une règle métier en Java.
* Gérer les cas d'erreurs via des exceptions.
* Valider le fonctionnement via des tests unitaires automatisés avec **JUnit 5**.

=== Environnement de travail
* **Dossier Source** : `src/main/java/com/pucesin/service/`
* **Dossier de Test** : `src/test/java/com/pucesin/service/` (à marquer comme *Test Sources Root* dans l'IDE).
* **Dépendance** : Vérifiez que `junit-jupiter-api` est bien présente dans votre `pom.xml`.

== Travail à réaliser (À coder)

=== Étape 1 : Création du Service d'Estimation
Créez une classe nommée `EstimationService`. Elle doit contenir une méthode `calculerPrixFinal` qui prend un objet `Livre` en paramètre et retourne un `float`.

**Règles de gestion à implémenter :**

* **Bonus Neuf** : Si l'état du livre est "neuf" (insensible à la casse), le prix de rachat doit être majoré de **10%**.
* **Cas Général** : Pour tout autre état, le prix reste inchangé.
* **Sécurité** : Si le prix de rachat initial du livre est inférieur à 0, la méthode doit lever une exception de type `IllegalArgumentException` avec un message explicite.

=== Étape 2 : Mise en place des Tests Unitaires
Créez une classe de test `EstimationServiceTest`. Vous devez coder trois méthodes de test pour couvrir tous les scénarios :

1.  **testCalculBonusNeuf** : Vérifie qu'un livre à 10€ passe bien à 11€ s'il est neuf.
2.  **testCalculSansBonus** : Vérifie qu'un livre à 10€ reste à 10€ s'il est dans un état "bon".
3.  **testPrixNegatif** : Vérifie que le système lève bien une exception si le prix est de -5€.

[TIP]
====
Pour tester une exception en JUnit 5, utilisez :
`assertThrows(IllegalArgumentException.class, () -> { service.methode(...); });`
====

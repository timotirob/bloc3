== Mission 1 : Gestion des prestations de remise en état (PREE)

=== Question 1.1 : Modification de la structure de la base de données
L'objectif est d'intégrer les PREE (Prestations de Remise en État) identifiées par l'expert. Un véhicule correspond à un dossier, et un dossier peut contenir plusieurs prestations.

**Solution par Schéma Relationnel (Optimisé pour étudiants) :**

* **PREE** ([.underline]#refDossier#, [.underline]#numOrdre#, libelle, description, photo, typePREE, elementCarrosserie, traitementPeinture, refPiece, qtePiece
** **refDossier** : Clé étrangère en référence à `Dossier(refDossier)`.
** **numOrdre** : Entier (1 à n) permettant d'identifier la prestation au sein du dossier.
** **typePREE** : Colonne permettant de distinguer 'Peinture' de 'Pièce'.

**Note pédagogique :** On utilise ici une "Table Unique" Bottom Up pour gérer l'héritage. Les colonnes spécifiques à la peinture seront à `NULL` pour une prestation de type pièce, et inversement. C'est la solution la plus simple à implémenter pour vous qui débutez en modélisation et en SQL. Pour ceux qui veulent l'état de l'art me demander mon PPT sur les façons d'implémenter l'héritage en SQL

== Mission 2 : Gestion des rendez-vous (C#)

=== Question 2.1 : Justification de la classe abstraite
La classe `Expertise` est **abstraite** car elle représente un concept générique. Dans le monde réel de RestiLoc, une expertise est soit un `RendezVousClient`, soit un `PoolGarage`. On ne peut pas créer une expertise "tout court". En programmation, cela permet de factoriser les attributs communs (date, immatriculation) tout en interdisant l'instanciation directe de la classe mère.

// 2.2 Ajouter une expertise de type Pool
public void AjouterExpertisePool(string dossier, DateTime dateHeure, string lieu, string adresse, string immat, string marque, string modele) {
    PoolGarage uneExp = new PoolGarage(dossier, dateHeure, lieu, adresse, immat, marque, modele);
    this.lesExpertises.Add(uneExp);
}

// 2.3 Getter du motif d'indisponibilité
public string GetMotif() {
    return this.motif;
}

// 2.4 Liste des expertises avec indisponibilité
public List<Expertise> LesExpertisesIndispos() {
    List<Expertise> lesIndispos = new List<Expertise>();
    foreach(Expertise e in this.lesExpertises) {
        // On vérifie si l'objet Indisponibilite existe pour cette expertise
        if (e.GetIndisponibilite() != null) {
            lesIndispos.Add(e);
        }
    }
    return lesIndispos;
}

// 2.5 Comptage par motif
public int NbIndisponibilites(string leMotif) {
    int nb = 0;
    foreach(Expertise e in this.LesExpertisesIndispos()) {
        if (e.GetIndisponibilite().GetMotif() == leMotif) {
            nb++;
        }
    }
    return nb;
}

== Mission 4 : Cybersécurité et RGPD

=== 4.1 Analyse DICP
* **Question 4.1.1 (Risques)** :
** **Disponibilité** : Panne du serveur Web empêchant les experts de voir leur planning. *Mesure* : Redondance des serveurs.
** **Confidentialité** : Un expert accède aux dossiers d'un autre expert. *Mesure* : Authentification forte et contrôle d'accès.
* **Question 4.1.2 (Photo)** : La photo vise la **Preuve** (Non-répudiation). Elle prouve l'état réel du véhicule pour éviter toute contestation du client lors de la facturation.

=== 4.2 Protection des données (RGPD)
* **Question 4.2.1** : Une donnée personnelle identifie une personne (Nom). Une donnée sensible concerne la vie privée profonde (Santé, Religion). Les données de la table `Expert` sont **personnelles** mais **non sensibles**.
* **Question 4.2.2** : Le **hachage** est une fonction irréversible. On préfère le hachage car même si la base est volée, les mots de passe ne peuvent pas être "déchiffrés". *Algorithme recommandé* : BCrypt ou Argon2.

=== 4.3 Sécurité applicative
* **Question 4.3.1** : Injection SQL à la ligne 11 du PHP : `$sql .= " WHERE idExpert = ".$id;`. L'identifiant est concaténé directement.
* **Question 4.3.2** : Faille **IDOR**. L'application demande `idExpert=2`. Un utilisateur peut modifier ce chiffre dans l'URL pour voir les missions d'un collègue car le serveur ne vérifie pas l'identité de celui qui demande.

Pour corriger l'IDOR, le serveur ne doit pas recevoir l'ID de l'expert en paramètre GET, mais le récupérer via un **Token de session sécurisé** généré au moment de la connexion.

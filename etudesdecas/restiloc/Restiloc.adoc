== Présentation du contexte

De plus en plus, particuliers et professionnels choisissent pour leur(s) véhicule(s) de signer un contrat de location avec une société de financement sur une période qui est le plus souvent fixée entre 12 et 48 mois et peut aller jusqu’à 60 mois. Ces contrats sont signés par le biais de garages liés à une ou plusieurs marques automobiles. À la fin du contrat, si le véhicule est restitué, il doit être expertisé afin de relever tous les dommages qu’il a subis durant la période de location. Cette expertise peut conduire à des frais de remise en état qui seront facturés au client par la société de financement.

=== L'organisation cliente
La société RestiLoc, implantée dans tous les pays d’Europe, est spécialisée dans l’expertise de véhicules en fin de contrat, pour le compte des sociétés de financement. La filiale RestiLoc France compte ainsi environ 100 experts, répartis sur tout le territoire français.

=== Le prestataire de services informatiques
La direction des systèmes d'information (DSI) de RestiLoc est chargée d'assurer la mise en place et la maintenance des infrastructures réseau ainsi que de développer les logiciels métier. La DSI a développé le logiciel **Torapix** permettant notamment aux experts d’accéder :

* à la planification de leurs missions d’expertise ;
* aux informations de chaque mission : société de financement, client, détails du véhicule.

Le logiciel Torapix s’appuie sur une base de données SQL Server hébergée par la DSI.

== Mission 1 – Gestion des prestations de remise en état (PREE)
(Document à utiliser : 1)

Dans le cadre de l'évolution du système d'information, il est demandé à la DSI de faire évoluer Torapix pour renseigner le détail des prestations de remise en état (PREE) identifiées par l’expert pour chaque dossier.

Un dossier d’expertise correspond à un seul véhicule. Ces prestations seront identifiées par la référence du dossier d’expertise suivie d'un numéro d'ordre (séquentiel de 1 à x). Chaque prestation sera décrite par un libellé et pourra être précisée par une description. Une photo est réalisée pour chaque prestation.

Il existe deux types de PREE :

* Les prestations **"peinture"** : concernent des travaux sur un ou plusieurs éléments de carrosserie (pare-chocs, aile, etc.), chacun nécessitant un traitement (voile, peinture partielle, etc.).
* Les prestations **"pièce"** : concernent le remplacement de pièces cassées. Il faut mémoriser les pièces à changer (libellé, référence) et la quantité.

**Question 1.1 :** Proposer une modification de la structure de la base de données pour intégrer la gestion des PREE ( diagramme de classes ou schéma relationnel).

=== Étape 2 : Service Web PHP Sécurisé (getLesMissionsByIdExpert.php)

Ce script utilise l'extension PDO avec le pilote MySQL. Les requêtes préparées neutralisent les injections SQL identifiées au Document 6[.

[source,php]
----
<?php
header('Content-Type: application/json');

// Configuration de la connexion
$host = 'localhost';
$db   = 'restiloc_db';
$user = 'admin_api';
$pass = 'P@ssword123!';
$charset = 'utf8mb4';

$dsn = "mysql:host=$host;dbname=$db;charset=$charset";

try {
    $bdd = new PDO($dsn, $user, $pass);
    $bdd->setAttribute(PDO::ATTR_ERRMODE, PDO::ERRMODE_EXCEPTION);

    // Récupération sécurisée de l'idExpert (Prévention IDOR et Injection)
    $idExpert = filter_input(INPUT_GET, 'idExpert', FILTER_VALIDATE_INT);

    if ($idExpert) {
        $sql = "SELECT M.idMission, M.heureDebut, G.ville, M.immatriculation, G.nom as nomGarage
                FROM MissionExpertise M
                JOIN Garage G ON M.idGarage = G.idGarage
                WHERE M.idExpert = :idExpert
                AND M.dateMission = :dateJour
                ORDER BY M.heureDebut ASC";

        $stmt = $bdd->prepare($sql);
        $stmt->execute([
            'idExpert' => $idExpert,
            'dateJour' => date('Y-m-d')
        ]);

        $resultat = $stmt->fetchAll(PDO::FETCH_ASSOC);
        echo json_encode($resultat);
    } else {
        http_response_code(400);
        echo json_encode(["erreur" => "Paramètre idExpert manquant ou invalide"]);
    }
} catch (Exception $e) {
    http_response_code(500);
    echo json_encode(["erreur" => "Erreur interne du serveur"]);
}
?>

**Classe `RestiClient.java` (Appel API) :**

Cette classe remplace la logique de `GestionURL.java` en utilisant `HttpClient`.

[source,java]
----
package fr.restiloc.service;

import java.net.URI;
import java.net.http.*;
import java.util.*;
import org.json.JSONArray;
import fr.restiloc.model.Mission;

public class RestiClient {
    private static final String URL_API = "http://localhost/api/getLesMissionsByIdExpert.php?idExpert=";

    public static List<Mission> fetchMissions(int idExpert) throws Exception {
        HttpClient client = HttpClient.newHttpClient();
        HttpRequest request = HttpRequest.newBuilder()
                .uri(URI.create(URL_API + idExpert)).build();

        HttpResponse<String> response = client.send(request, HttpResponse.BodyHandlers.ofString());

        List<Mission> missions = new ArrayList<>();
        JSONArray jsonArray = new JSONArray(response.body());
        for (int i = 0; i < jsonArray.length(); i++) {
            var obj = jsonArray.getJSONObject(i);
            missions.add(new Mission(
                obj.getInt("idMission"), obj.getString("heureDebut"),
                obj.getString("ville"), obj.getString("immatriculation"),
                obj.getString("nomGarage")
            ));
        }
        return missions;
    }
}

== Mission 6 : Challenges Expertise & Cybersécurité (Approfondissement)

Cette section propose des exercices pratiques pour valider vos compétences en développement sécurisé et en logique métier.

=== Challenge 1 : La Faille IDOR (Insecure Direct Object Reference)
**Objectif :** Comprendre pourquoi on ne doit jamais faire confiance aux paramètres envoyés par le client (navigateur ou application).

**Énoncé :** Actuellement, votre client JavaFX demande les missions de l'expert n°1.

1. Modifiez manuellement le code Java pour demander l'expert n°2.
2. Que constatez-vous ?
3. Pourquoi est-ce un problème de sécurité majeur pour Restiloc ?

.Indice (HINT) - Cliquez pour dérouler
[%collapsible]
====
* **Où regarder ?** Dans la classe `RestiClient.java`, modifiez la variable `URL_API` ou l'appel à `fetchMissions(1)`.
* **Concept Cyber :** Une faille IDOR survient quand un utilisateur peut accéder à des données qui ne lui appartiennent pas en changeant simplement un identifiant dans l'URL.
* **Solution attendue :** Il doit être mentionné qu'un expert malveillant pourrait espionner ses collègues. La solution réelle serait d'utiliser un **Token de session** (jeton) généré après connexion.
====

=== Challenge 2 : L'audit de suppression (Trigger SQL)
**Objectif :** Garantir l'intégrité et la preuve (critères I et P du DICP) en cas de suppression de données.

**Énoncé :** On soupçonne certains experts de supprimer des missions difficiles pour alléger leur planning.

1. Implémentez le déclencheur (Trigger) `trig_audit_delete_mission` fourni dans l'annexe technique.
2. Supprimez une mission via une requête SQL : `DELETE FROM MissionExpertise WHERE idMission = 1;`.
3. Vérifiez que la table `AUDIT_SUPPRESSION` a bien enregistré la trace.

.Indice (HINT) - Cliquez pour dérouler
[%collapsible]
====
* **Le Trigger :** C'est un programme qui s'exécute automatiquement sur le serveur. Ici, il surveille l'événement `DELETE`.
* **Table 'OLD' (MySQL) :** Lors d'une suppression, MySQL stocke temporairement la ligne supprimée dans une table virtuelle nommée `OLD`. C'est là que le trigger récupère l'ID pour l'archiver.
* **Vérification :** Faites un `SELECT * FROM AUDIT_SUPPRESSION;` pour prouver que la trace existe.
====

=== Challenge 3 : Logique métier - Gestion des "Indispos"
**Objectif :** Manipuler le modèle de données et l'interface JavaFX.

**Énoncé :** Si un expert se rend sur place mais que le client est absent, la mission doit être marquée comme "Indisponible".

1. Ajoutez un bouton "Marquer Indisponible" sous votre tableau JavaFX.
2. Lorsqu'on clique dessus, la mission sélectionnée doit s'afficher en **Rouge** dans la liste (ou afficher un message "Indisponible").

.Indice (HINT) - Cliquez pour dérouler
[%collapsible]
====
* **Étape 1 :** Dans `Mission.java`, ajoutez un attribut `boolean indisponible` (par défaut à `false`) et son getter/setter.
* **Étape 2 (Contrôleur) :** Récupérez la ligne sélectionnée avec `tableMissions.getSelectionModel().getSelectedItem()`.
* **Étape 3 (UI) :** Pour changer la couleur, il faut utiliser une `setRowFactory` sur la TableView. Pour les moins avancés en Dév, commencez par simplement changer le texte d'un Label : `lblStatut.setText("Mission " + id + " indisponible");`.
====

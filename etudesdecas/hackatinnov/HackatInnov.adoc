La structure *Hackat'Innov* organise des événements de type Hackathon. Pour gérer ces événements, elle dispose d'une application interne nommée **Hackat'Orga**.

== Dossier A – Gestion des participants
L'objectif est d'étendre la base de données actuelle pour gérer le système de vote final et l'inscription des participants.

=== Mission A.1 – Évolution de la base de données pour la gestion des hackathons

==== Contexte de la mission
L’application Web Hackat’Orga permet actuellement de gérer uniquement l’étape d’initialisation d’un hackathon. Il s'agit de la faire évoluer pour y intégrer les étapes de publication et d’inscription des participants.

Le schéma de la base de données actuelle est fourni dans le **Document 1**. Il doit évoluer pour intégrer les données utiles à l’inscription en ligne des participants et à l’édition du planning détaillé des hackathons.

==== Cahier des charges des évolutions

* **Inscription des participants :**
- Pour chaque hackathon : nombre de places limité et date limite d'inscription.
- Profil membre : nom, prénom, mél, téléphone, date de naissance et lien vers un portfolio.
- Processus : sélection du hackathon, saisie d'un texte libre (compétences/outils).
- Stockage : mémorisation de la date de saisie et attribution d'un numéro unique d'inscription séquentiel au sein de chaque hackathon.

* **Constitution des équipes :**
- Un inscrit ne peut être affecté qu'à une seule équipe.
- Une équipe travaille sur un seul projet et possède un nom d'équipe.
- Identification : les équipes sont identifiées relativement au projet choisi.
- Responsabilité : un chef de projet est défini dans chaque équipe parmi ses participants.

* **Vote des membres du jury :**
- Fourniture d'un lien vers le prototype par chaque équipe.
- Chaque membre du jury attribue une note de 0 à 5 pour chaque équipe.
- Un membre du jury peut être participant, mais uniquement pour d'autres hackathons.

[NOTE]
Une équipe appartient à un seul Hackathon. Un participant appartient à une seule équipe.

[IMPORTANT]
====

==== Contexte et enjeux
L’ordre de déroulement des phases (accueil des participants, phases de travail, repas, présentation des projets, délibération du jury et proclamation des résultats) est identique pour tous les hackathons. Pour chaque hackathon, les organisateurs doivent pouvoir éditer le planning détaillé afin que les participants puissent en prendre connaissance dès leur inscription.

=== 5.1. Création de la base de données (DDL)

Exécutez le script suivant pour générer la structure complète de la base `hackatinnov`. Ce script intègre l'ensemble des entités vues dans les missions précédentes.

[source,sql]
----
CREATE DATABASE IF NOT EXISTS hackatinnov DEFAULT CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE hackatinnov;

CREATE TABLE ORGANISATEUR (
    id INT AUTO_INCREMENT PRIMARY KEY,
    statut VARCHAR(50),
    nom VARCHAR(100) NOT NULL,
    siteWeb VARCHAR(255),
    mel VARCHAR(150) NOT NULL
);

CREATE TABLE MEMBRE (
    id INT AUTO_INCREMENT PRIMARY KEY,
    nom VARCHAR(50) NOT NULL,
    prenom VARCHAR(50) NOT NULL,
    mel VARCHAR(150) NOT NULL UNIQUE,
    telephone VARCHAR(20)
);

CREATE TABLE HACKATHON (
    id INT AUTO_INCREMENT PRIMARY KEY,
    dateHeureDebut DATETIME NOT NULL,
    dateHeureFin DATETIME NOT NULL,
    lieu VARCHAR(100),
    ville VARCHAR(50) NOT NULL,
    theme VARCHAR(150) NOT NULL,
    affiche VARCHAR(255),
    objectifs TEXT
);

CREATE TABLE ORGANISER (
    idHackathon INT,
    idOrganisateur INT,
    PRIMARY KEY (idHackathon, idOrganisateur),
    FOREIGN KEY (idHackathon) REFERENCES HACKATHON(id) ON DELETE CASCADE,
    FOREIGN KEY (idOrganisateur) REFERENCES ORGANISATEUR(id) ON DELETE CASCADE
);

CREATE TABLE EVENEMENT (
    id INT AUTO_INCREMENT PRIMARY KEY,
    idHackathon INT NOT NULL,
    idAnimateur INT NOT NULL,
    libelle VARCHAR(150) NOT NULL,
    dateHeure DATETIME NOT NULL,
    duree INT NOT NULL,
    salle VARCHAR(50),
    typePublic VARCHAR(100),
    FOREIGN KEY (idHackathon) REFERENCES HACKATHON(id) ON DELETE CASCADE,
    FOREIGN KEY (idAnimateur) REFERENCES MEMBRE(id)
);

CREATE TABLE CONFERENCE (
    idEvenementConf INT PRIMARY KEY,
    theme VARCHAR(150) NOT NULL,
    FOREIGN KEY (idEvenementConf) REFERENCES EVENEMENT(id) ON DELETE CASCADE
);

CREATE TABLE INITIATION (
    idEvenementInit INT PRIMARY KEY,
    nbPlaces INT NOT NULL,
    FOREIGN KEY (idEvenementInit) REFERENCES EVENEMENT(id) ON DELETE CASCADE
);

CREATE TABLE INSCRIRE (
    idEvenementInit INT,
    idMembre INT,
    PRIMARY KEY (idEvenementInit, idMembre),
    FOREIGN KEY (idEvenementInit) REFERENCES INITIATION(idEvenementInit) ON DELETE CASCADE,
    FOREIGN KEY (idMembre) REFERENCES MEMBRE(id) ON DELETE CASCADE
);

CREATE TABLE PHASE (
    id INT AUTO_INCREMENT PRIMARY KEY,
    libelle VARCHAR(100) NOT NULL
);

CREATE TABLE PLANNING (
    idHackathon INT,
    dateHeureDebut DATETIME,
    idPhase INT NOT NULL,
    duree INT NOT NULL,
    PRIMARY KEY (idHackathon, dateHeureDebut),
    FOREIGN KEY (idHackathon) REFERENCES HACKATHON(id) ON DELETE CASCADE,
    FOREIGN KEY (idPhase) REFERENCES PHASE(id)
);
----

Créez les deux fichiers suivants dans votre projet pour exposer l'API sécurisée au format JSON, attendue par l'application mobile (Vue.js).

**Fichier : `util/bdd.php`**
[source,php]
----
<?php
declare(strict_types=1);

function connexionBdd(): PDO {
    $host = '127.0.0.1';
    $db   = 'hackatinnov';
    $user = 'root'; // À adapter selon votre configuration (ex: Laragon = root / sans mot de passe)
    $pass = '';
    $charset = 'utf8mb4';

    $dsn = "mysql:host=$host;dbname=$db;charset=$charset";
    $options = [
        PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION,
        PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,
        PDO::ATTR_EMULATE_PREPARES   => false,
    ];

    try {
        return new PDO($dsn, $user, $pass, $options);
    } catch (\PDOException $e) {
        throw new \PDOException($e->getMessage(), (int)$e->getCode());
    }
}
?>
----
